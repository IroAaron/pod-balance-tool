# PoD Balance Tool

Сайт-справочник по контенту и балансу для **PoD** — карточного рогалика на Godot (Steam). Статический React-сайт (без бэкенда), который затягивает игровые данные из Google Sheets/Apps Script/CSV, даёт браузер предметов и позволяет вручную собирать «билды» — тематические группы предметов, которые усиливают друг друга.

Полное ТЗ (на русском): `PoD_Web_Balance_Prompt.md` (лежит вне репозитория, у пользователя в Загрузках — путь может отличаться). Оно во многом устарело после знакомства с реальными таблицами — актуальная схема данных описана ниже.

## Быстрый старт

```bash
npm install
npm run dev      # http://localhost:5173
npm run build    # tsc -b && vite build (+ пересобирает манифест спрайтов, см. ниже)
npm run lint
npm test         # vitest — пока только src/core/domain/relations.test.ts (каскадная автосборка)
```

Развёртывание — GitHub Pages (статика, `dist/`), бэкенда нет. Все правки пользователя (билды, иконки, кастомные значения) живут в `localStorage` браузера + экспорт/импорт JSON-снапшота на странице «Источники».

## Что здесь есть (страницы)

- **Источники** (`/sources`) — два поля-ссылки (конфиг / переводы: Google Sheets CSV-ссылка ИЛИ URL Apps Script Web App, см. ниже) + ручная загрузка CSV. Показывает отчёт о распознанных таблицах и предупреждения.
- **Предметы** (`/items`, `/items/:id`) — список с поиском/фильтром по тегу и типу/сортировкой; мини-карточка в списке показывает спрайт/иконку, имя, id, **описание из переводов** (обрезано до 3 строк через `-webkit-line-clamp`, если есть) и теги. Карточка предмета (`/items/:id`): спрайт/иконка, имя, описание (полностью), теги, **раздел «Параметры»** (сырая строка предмета из его исходной таблицы — `item.raw` целиком, минус `ItemId`/`Tags`, которые уже показаны отдельно, и пустые ячейки; для Card/House/Artefact набор колонок разный и берётся автоматически — никакого хардкода по категориям не нужно, просто `item.raw` уже содержит колонки той таблицы, откуда пришла строка), механики, цепочка прокачки, замены (ReplaceItem/ReplaceOnTrigger), панель «Возможно связано».
- **Билды** (`/builds`, `/builds/:id`) — список карточек: заголовок (иконка+название, клик → карточка билда), под ним — кликабельные спрайты предметов (раздел «Возможно связано с» убран с карточек списка 2026-07-19 — остался только на `/builds/:id`, см. ниже). Поиск/фильтр по тегу/фильтр по типу билда (жёстко зашитый список `Card`/`House`/`Artefact` в `BuildsPage.tsx` — не `store.paramValues.ItemType`, который также тянет `TargetType`/`BonusTargetType` из механик и замусорил бы список; по типу **первого/корневого** предмета билда, не любого; «Выбрать все»/«Снять все»)/сортировка/галочка «Учитывать прокачки (+/++)» (по умолчанию выключена — оба алгоритма ниже игнорируют уровни прокачки, пока не включишь)/кнопки «Предложить билды» (автокластеризация по тегам/id) и «Собрать билды от очков» (каскад от PlayerScore, см. ниже)/«+ Создать билд». `/builds/:id` — форма редактирования (имя/иконка/описание/состав/**ручные связи с другими билдами**).
- **Граф** (`/graph`) — force-directed граф в стиле Obsidian, но **узлы — билды, а не предметы** (второй заход: первая версия графа была Item+Build нодами, её убирали и возвращали дважды — см. историю ниже, если снова придётся переделывать). Билд связан с билдом, если у них есть общие предметы (толщина линии — сила связи, плюс цвет от красного к зелёному, чем сильнее) или если связь добавлена вручную (оранжевая линия). Иконка узла — то же самое, что и на карточке `/builds` (спрайт/эмодзи корневого предмета, см. `resolveBuildIcon`/`BuildIcon` выше), только через собственный `Image()`-кэш вместо `<img>`, т.к. canvas не умеет `onError`. Дропдаун «Предметы» (мультиселект `Card`/`House`/`Artefact`, по умолчанию все включены, тот же хардкод-список что и `BUILD_TYPE_OPTIONS` на `/builds`) фильтрует узлы по типу **корневого** предмета билда — билды без определяемого типа (пустой состав) фильтром не скрываются. Клик по узлу → карточка билда.
- **Аналитика** (`/analytics`) — сейчас только раздел «неиспользуемые спрайты» (файлы в `public/pod-mini-characters/`, на которые не ссылается ни один загруженный предмет).

## Архитектура

- React 19 + Vite + MUI v9 (⚠️ **не v5/v6** — `Stack` больше не принимает `alignItems`/`justifyContent`/`flexWrap`/`gap` напрямую как пропсы, только через `sx`, иначе TS-ошибка) + react-router.
- `src/core/GameStore.ts` — рукописный mutable-класс-синглтон (не Redux/Zustand). Реактивность — через собственный pub/sub: счётчик `version` + `notify()` после каждой мутации, `useStore()` (`src/ui/hooks/useStore.ts`) подписывается через `useSyncExternalStore`. **Если добавляете новый метод-мутатор в GameStore — не забудьте вызвать `this.notify()` в конце, иначе компоненты не перерисуются.**
- `store.items` — **геттер**, не поле: фильтрует `store.allItems` (весь импортированный набор) по наличию перевода (см. «Правила видимости» ниже). Для «сырых», непрофильтрованных данных используйте `store.allItems`.
- Персистентность — `localStorage`, ключи `pod-balance-tool:v1:*` (`src/core/persistence/localStore.ts`), плюс экспорт/импорт JSON.
- `src/core/import/` — пайплайн импорта: `csv.ts` (papaparse), `sheetSource.ts` (fetch Google Sheets CSV или Apps Script JSON), `sanitize.ts` (чистка «строк-комментариев», см. ниже), `tableClassifier.ts` (определяет тип таблицы по заголовкам колонок), `tableNames.ts` (сопоставление имени вкладки), `normalize.ts` (сырые строки → доменные модели).
- `src/core/domain/` — доменная логика без UI: `mechanicTables.ts` (структура колонок механик), `paramRegistry.ts` (динамический список значений параметров), `relations.ts` (связи предметов/билдов — сердце «Возможно связано» и автосборки билдов), `sprites.ts` (путь к спрайту).
- **Принцип «никаких зашитых констант»**: ни одно допустимое значение параметра (ItemType/TargetColor/Place/ActivatorType/ItemTag и т.д.) не хардкодится в коде — все выводится из загруженных данных (`paramRegistry.ts`) + ручные кастомные значения пользователя. Зашиты только *имена* колонок (структура), не значения.

## Реальная схема данных (важно!)

ТЗ описывало упрощённые примеры — реальные таблицы (проверено на живом экспорте пользователя) устроены иначе:

| Таблица | Что это | Особенности |
|---|---|---|
| `Cards` / `Houses` / `Artefacts` | Предметы, **три отдельные таблицы**, не одна | Нет колонки `ItemType` — категория (`Card`/`House`/`Artefact`) выводится из **имени вкладки**, не из данных (`ITEM_CATEGORY_HINTS` в `normalize.ts`) |
| `item_name` / `item_desc` | Переводы | Формат `key,ru,en` (не `key,value`). `item_desc`: ключ = `{id}_desc` |
| `Enums` | Справочник допустимых значений параметров | **Широкий/рваный формат** — одна колонка = одно измерение (ItemType/TargetColor/.../ItemTag), независимые списки без связи между строками. НЕ строка-на-запись |
| `MechActivate`, `MechAddValue`, `MechChangeColor` | Механики (Activator→Target фильтры) | В целом совпадают с ТЗ, плюс реальные доп.колонки `UseActivatorIds`/`MyPositionReq`/`Chance` |
| `MechAddItem` | Спавн/замена предмета | В ТЗ было не дописано; реальные колонки: `...,ItemMech,NewItemId,CopiedTargetType,...` (`ItemMech` = `поставить`/`удалить`) |
| `MechAddTag` | Выдача/снятие тега | В ТЗ было не дописано; реальные колонки: `...,TagMech,NewTags,TagsCount` |
| `ReplaceItem` | Условная замена предмета | `ItemIdToReplace,ReplacementItem,NeededItem,NeededItemPlace,NeededItemNumber` — именно этот механизм стоит за примером «Продюсер создаёт Рок музыканта из Уличного музыканта» |
| `ReplaceOnTrigger` | Замена по таймеру/триггеру | `ItemIdToReplace,ReplacementItem,ReplacementItemsTagForName,OnballStop,DurationType,Duration` |
| `CardUpgrades` | Цепочки прокачки | `UpgradeChainId,UpgradeId1,UpgradeId2,UpgradeId3...` (число тиров не фиксировано, ищем по regex `UpgradeId\d+`) |

Классификация таблиц — по **сигнатуре колонок**, не по имени вкладки (см. `tableClassifier.ts`), потому что имя вкладки ненадёжно (см. ниже).

### Реальные грабли, на которые уже наступали

- **Имена вкладок ≠ имена файлов.** CSV, выгруженный из Google Sheets, называется `"<Название таблицы> - <ИмяВкладки>.csv"`, а Apps Script (`sheet.getName()`) отдаёт просто `"ИмяВкладки"`. Сопоставление имени — только через `tableNameOf()`/`matchesTableName()` (`tableNames.ts`), которые берут последний сегмент после `" - "`.
- **Строки-комментарии выдают себя за данные.** Во второй строке нескольких таблиц (`MechAddValue`, `MechAddTag`, `MechChangeColor`, `ReplaceOnTrigger`) — описание колонки на русском, иногда прямо в колонке Id. Отличаем от настоящих коротких кириллических значений (`поставить`/`удалить`/`дать`) по числу слов (`sanitize.ts`: 3+ слова с кириллицей → считаем комментарием и затираем).
- **Числа из Apps Script — не строки.** `getValues()` отдаёт реальные JS `number`/`boolean` для числовых ячеек, а не строки. Раньше это ронял весь сайт (`.trim()` на числе). Чинится один раз в `sheetSource.ts` (`cellToString`/`stringifyRow`) — единственное место, где вообще может прийти не-строка.
- **Разреженная первая строка может отравить классификацию таблицы.** `fetchAppsScriptJson` раньше брал `headers` только из `Object.keys(rows[0])` — если у первой строки не хватало какого-то ключа, колонка пропадала из `headers` для всей таблицы (актуально только для источника Apps Script — CSV-загрузка всегда читает фиксированную строку заголовков через `papaparse`). Починено: `headers` теперь — объединение ключей по всем строкам, а не только по первой. Оказалось, что это НЕ было настоящей причиной пропавших описаний — см. следующий пункт.
- **Настоящая причина «описаний нет вообще ни у одного предмета» — `sanitizeParsedTable` стирал их сам.** Эвристика «3+ кириллических слова в ячейке = строка-комментарий, затираем» (см. ниже) раньше применялась ко **всем** таблицам без исключения, включая `Translations`. Реальное описание предмета — это ровно то, что эвристика ищет и стирает (многословная кириллическая проза), так что `item_desc` тихо превращался в пустые строки почти везде, кроме коротких описаний (≤2 слова), которые случайно выживали — из-за этого казалось, что баг «через один» предмет, а на деле рвался весь столбец. Ключ перевода при этом существовал и совпадал правильно — `store.itemDescription()` возвращал `""` (не `undefined`), и UI корректно это скрывал, то есть весь код ниже по цепочке работал верно против уже испорченных данных. Починено: `ImportService.buildResult` теперь классифицирует таблицу **до** санитизации (классификация читает только `headers`, не `rows`, так что порядок можно менять безопасно) и применяет `sanitizeParsedTable` только к НЕ-`Translations` таблицам. Проверено на реальных `item_name.csv`/`item_desc.csv`/`PoD_config.zip` от пользователя: до фикса `c_chel_money_for_activation_chel_near_1_desc` резолвился в `""`, после — в полный текст; скриптовая проверка по всем 180 реальным `Cards`-предметам после фикса не нашла ни одного с пустым описанием. Санитизация комментарий-строк механик (ради чего эта эвристика вообще существует) при этом не пострадала — перепроверено на реальном `MechAddValue.csv`.
- **Подстрочный поиск колонки — опасен.** Fallback-поиск колонки по подстроке `"type"` однажды поймал `ValueUsageType` вместо настоящего типа предмета (`ItemType` не было, а `*Type`-колонки — есть всегда). Урок: подстрочные fallback'и — только когда точное совпадение действительно недоступно, и проверять на коллизии с соседними `*Type`/`*Tag` колонками.
- **Несовпадение id между конфигом и переводами.** ~40 предметов из конфига не находили перевод напрямую — у 38 из них в конфиге не хватало префикса `standart_`, который есть в ключе перевода (например `in_a_money_per_red_loop` vs `standart_in_a_money_per_red_loop`). Это НЕ починено эвристикой (слишком ненадёжно угадывать по паттерну id) — предметы без перевода просто скрываются (см. ниже), пользователю показано как факт, ждёт решения на стороне таблиц.

## Правила видимости и очистки данных

- **Предмет без перевода — скрыт везде.** Если для `item.nameKey` нет записи в `translations`, предмет не попадает в `store.items` (геттер, фильтрует `store.allItems`). Иначе — везде.
- **Уровни прокачки (+/++) — не самостоятельные карточки в списках связей/графа.** `higherTierIds(upgradeChains)` (`relations.ts`) — общая утилита, вычисляет все id-тиры кроме первого в каждой цепочке `CardUpgrades`; используется и в билд-карточках, и в «Возможно связано». Предмет всё ещё существует и доступен напрямую (`/items/:id`), просто не всплывает как отдельная подсказка.
  - ⚠️ Работает только для того, что реально прописано в `CardUpgrades` — если у предмета нет записи там (даже если имя в переводах явно "Х+"), фильтр его не поймает. Известный пробел: Черлидер/Болельщик не зарегистрированы в `CardUpgrades`, хотя явно апгрейдятся. Этот пробел **всё ещё актуален** для графа и «Возможно связано» — но не для генерации билдов, см. ниже.
  - Для **генерации билдов** (`GameStore.itemsForBuildGeneration`) пробел закрыт отдельно: предмет исключается, если он либо в `higherTierIds`, либо его переведённое название заканчивается на `+`/`++` (`/\+{1,2}$/`) — так что незарегистрированные в `CardUpgrades` тиры тоже не просачиваются в черновики, пока не включена галочка «Учитывать прокачки».

## Модель «билда» (для автоподсказок)

Билд — это не просто «общий тег». Настоящая механика (по разобранным вместе с пользователем примерам, реальные id — см. `src/core/domain/relations.ts` и историю чата):

**payoff-предмет** слушает событие/фильтр (свой `ActivatorType` + `ActivatorTag`/`ActivatorColor`/`Place`) — **feeder-предметы** производят ровно это условие, часто из совсем другой таблицы механик, без общего тега вовсе. Пример: Черлидер/Болельщик (`MechChangeColor` — перекраска) кормят Художника (`MechAddValue`, `ActivatorType=ColorChange, Place=All` — слушает любую смену цвета где угодно).

⚠️ **2026-07-17: связи только по механикам, теги как сигнал полностью убраны** — ни в автосборке билдов, ни в панели «Возможно связано» на карточке предмета простое «у этих двух предметов совпадает тег» больше не считается связью. `computeTagSets`/`MECHANIC_TAG_FIELDS` удалены из `relations.ts` целиком.

Сигналы, которые реально участвуют в автосборке билдов (`computeSuggestedBuilds`) — «сильные», объединяют предметы в один черновик:
1. Прямая ссылка по Id в любом поле механики (генерический скан значений против известных id)
2. Общая цепочка прокачки (`CardUpgrades`)
3. Связь через `ReplaceItem`/`ReplaceOnTrigger`

Сигнал «производит событие → слушает событие» (`producedActivatorType`/`computeListenedEvents`, например `MechChangeColor` → `ColorChange`) — **сознательно НЕ участвует** в автосборке, только в информационных панелях «Возможно связано»/«Возможно связано с».

⚠️ Не путать с тег-фильтрами внутри каскадной автосборки (`computeCascadeBuilds`, уровни 2/3/5 — `BonusTargetTag`/`ActivatorTag`/`TargetTag`, см. ниже) — это **не** тот же сигнал. Там тег — это конкретное поле-фильтр одной механической строки, направленная причинная связь, а не симметричное «у двух предметов пересекаются наборы тегов». Каскад читает `item.tags` напрямую и никогда не использовал `computeTagSets`, поэтому убран не был.

Транзитивную композицию билдов (билд A кормит билд B через то, что payoff билда A — фидер в билде B) — сознательно не автоматизировали, слишком нечёткая логика для авто-эвристики.

## Каскадная автосборка от PlayerScore (`computeCascadeBuilds`, `relations.ts`)

Второй, принципиально другой генератор черновиков — кнопка «Собрать билды от очков» на `/builds`, рядом с «Предложить билды» (сосуществуют, один билд может появиться из обоих алгоритмов).

⚠️ Этот алгоритм переписан один раз в тот же день — первая версия (безлимитный BFS-каскад наружу) заменена на фиксированную неглубокую структуру по прямой просьбе пользователя («Не нужно делать долгие каскады»). Текущая версия — **ровно 5 уровней, без рекурсии дальше пятого**.

- **Корень** — любой предмет с `MechAddValue`-строкой, где `TargetType === "PlayerScore"` (буквальная строка, захардкожена намеренно — структурный маркер, тот же прецедент что `"ColorChange"`/`"поставить"`/`"удалить"` в `producedActivatorType`), **и** `TargetValueType === "MainValue"`, **и** у предмета настоящий диапазон `ValueMin`/`ValueMax` (не оба равны 0 или пусты) — по умолчанию.
  - ⚠️ **История вопроса — сюда стоит заглянуть, прежде чем снова трогать этот код.** Было 4 попытки: (1) просто `TargetValueType=MainValue` — недостаточно; (2) эвристика `hasThematicHook` (`ActivatorTag`/`ActivatorColor`/`Bonus*`/особые события) — придумана по 3 примерам, сломалась на «Продюсере»; (3) проверка `ValueMin`/`ValueMax` предмета (текущее правило) — идея пользователя, но сломалась на «Продюсере» тоже (`ValueMin=5, ValueMax=5`, не ноль); (4) проверка текста описания предмета на `"ValueOrRange"` — самый точный вариант (ноль ложных срабатываний на всех 143 реальных строках), но пользователь явно попросил вернуться к **структурной** проверке (`ValueMin`/`ValueMax`) вместо анализа текста описания — «нужно будет все-таки сделать по механике». **Текущее правило — снова (3), сознательно, по прямому запросу пользователя.**
  - `ValueMin`/`ValueMax` теперь настоящие типизированные поля `Item` (`item.valueMin`/`item.valueMax: number | undefined`), парсятся в `normalizeItemsTable` (`normalize.ts`) через новый `parseOptionalNumber` (пустая ячейка → `undefined`, поддержка запятой как десятичного разделителя — в реальных данных встречается `"1,01"`), а не читаются из `item.raw` на лету.
  - ⚠️ **Осознанно принятое ограничение**: «Продюсер» (`c_chel_money_2_1`, `ValueMin=5, ValueMax=5`) **по-прежнему проходит** и становится корнем, хотя его описание («Дает ${MoneyValue}», без `ValueOrRange`) доказывает, что он такой же «плоский», как и правильно исключённые примеры. Это не баг — пользователь явно выбрал структурное правило вместо текстового, зная про этот случай. **Не «чинить» это молча** — если пользователь снова спросит про Продюсера, предложить объединить оба сигнала (ValueMin/ValueMax И текст описания), а не тихо переключаться туда-обратно.
  - Ровно 1 из 144 реальных PlayerScore-строк использует `TargetValueType=MoneyValue` (`c_start_chel_1`, стартовый персонаж) — тоже исключён по умолчанию, той же галочкой «Показывать билды, с MoneyValue?» (`includeMoneyValueRoots`, по умолчанию выключена).
  - `vitest` (`npm test`) и `src/core/domain/relations.test.ts` — юнит-тесты на синтетических фикстурах с реальными проверенными примерами, включая тест с говорящим названием «KNOWN ACCEPTED GAP», который явно проверяет, что Продюсер-подобные предметы **проходят** — это документирует компромисс, а не баг, который тест не замечает.
  На каждый корень — один черновик.
- **Уровень 1 — корень.** Предмет, который зарабатывает PlayerScore; ради него существует весь билд.
- **Уровень 2 — скейлеры.** Предметы, подходящие под Bonus-фильтр строки-payoff, через `collectScalers` — **пересечение**, а не объединение: если у строки заданы и `BonusTargetTag`, и `BonusTargetType`, предмет должен совпадать по **обоим** сразу (одно составное условие «Card с тегом Rich», а не «что угодно с тегом Rich» ИЛИ «любой Card» по отдельности — второе на реальных данных подключало почти весь пул предметов, т.к. `BonusTargetType=Card` почти ничего не фильтрует). `type` учитывается только если это реальная категория предмета (`Card`/`House`/`Artefact`) — значения вроде `Road`/`NotRoad`/`All` описывают ячейку поля, а не `item.itemType`, и просто игнорируются как фильтр типа. Сверяется со **статическими** `item.tags`/`item.itemType` (не с «раздутым» набором тегов из `computeTagSets`, который используется в других местах — здесь уровень 2 означает «уже подходит под профиль», а не «производит профиль», это уровень 5).
- **Уровень 3 — активаторы корня.** `UseActivatorIds`, если строка называет конкретный предмет; иначе — предметы, подходящие под `ActivatorTag`/`ActivatorColor`, плюс структурно производящие нужное событие `ActivatorType` (`producedActivatorType`) — если у строки задан `ActivatorTag`, событийное совпадение сужается до продюсеров этого события **именно для предмета с этим тегом** (`itemIdsByProducedTaggedEvent`, ключ `"событие|тег"`), а не любого продюсера события вообще (иначе, например, Дурка с `ActivatorType=ItemPlaced, ActivatorTag=Crazy` подключала вообще любой домик, что-то ставящий через `MechAddItem`, а не только тех, кто ставит именно Crazy-тегированный предмет).
- **Цвет (`ActivatorColor`/`BonusTargetColor` на уровнях 3 и 5).** `Same`/`NotSame`/`Random` — не конкретные цвета, а вычисляемые в рантайме плейсхолдеры (`RELATIVE_COLOR_VALUES`); сравнивать их строкой с `MechChangeColor.NewColor` бессмысленно что в одну, что в другую сторону. Если цветовое поле payoff'а — один из этих трёх, в матч идут **все** перекрасчики (`recolorersForColor`/`allRecolorers`), не только те, чей `NewColor` буквально совпадает строкой — та же логика, что уже применяется к `ActivatorType=ColorChange`-событию.
- **Уровень 4 — спавнеры корня.** Предметы, ставящие его через `MechAddItem` (`ItemMech="поставить"`), либо `itemIdToReplace`-сторона правила `ReplaceItem`/`ReplaceOnTrigger`, которое в него превращается. ⚠️ **Направленно**, не симметрично: `replacementItem` не считается «спавнером» своего `itemIdToReplace` — это то, во что предмет превращается, а не то, что его производит (в отличие от `buildReplaceMates` в `relatedItems`, где связь намеренно симметричная — там направление не имеет значения).
- **Уровень 5 — спавнеры уровня-2-скейлеров и уровня-3-активаторов**, плюс: перекрасчики, подходящие под `BonusTargetColor` payoff-строки (с тем же relative-цвет правилом, что и уровень 3); те, кто выдаёт тег через `MechAddTag.NewTags`, совпадающий с `BonusTargetTag`/`ActivatorTag`; и всё, что **усиливает** корень — либо напрямую по id (`targetersOf`), либо через `TargetTag`-фильтр, совпадающий с одним из собственных статических тегов корня (`itemIdsByTargetedTag`, «меняет Value первого», например предмет, поднимающий Value любой ближней карты с тегом Sport). ⚠️ Оба этих сигнала (`targetersOf`/`itemIdsByTargetedTag`) строятся **только из строк `MechAddValue`** — только они реально поднимают Value/MoneyValue/etc цели; строка из любой другой таблицы, которая просто *называет* id/тег корня (убивает его через `MechAddItem`, перекрашивает, меняет тег), не читается как «усиление», даже если тег совпал (реальный кейс: у Рок музыканта тег `Rich` — флейвор, не связанный с его Music-темой payoff'а; Дешёвый мотель тоже фильтрует по `TargetTag=Rich`, но это `MechAddItem`-спавн Маньяка, а не буст — раньше совпадение тега затягивало его в билд Рок музыканта). Матчинг тега — не `TargetType`/`itemType`: `TargetType` почти всегда `"Card"` (подавляющее большинство предметов), так что матчинг по типу тут же подключил бы почти всё ко всему; проверено и отклонено на синтетическом негативном контроле при реализации.
- «Хорошее» удаление рядом с корнем (например, Медсестра зарабатывает, когда рядом кого-то убили) — это **не** taргетинг-по-id/тегу выше, а отдельный, уже направленный сигнал: корень **слушает** `ActivatorType=ItemRemoved` (уровень 3, событийный путь), кто-то другой **производит** это событие. Направление здесь жёсткое по построению, отдельно чинить не потребовалось.
- Каждый уровень считается напрямую от корня/уровня-2/уровня-3 — уровни 4/5 **не** порождают собственную дальнейшую рекурсию (кроме прямого учёта уровня-3-активаторов на уровне 5, см. выше).
- **Название**: `Билд от {имя корневого предмета}` (в отличие от `computeSuggestedBuilds`, где имя черновика пустое). Иконку черновик больше не «запекает» на момент генерации — она резолвится живьём при каждом рендере, см. `BuildIcon` ниже.
- **2026-07-18: точечная правка по 7 реальным жалобам пользователя** (Бухгалтер/Военный/Казино/Дурка/Картина/Музей/Рок музыкант) — все свелись к 6 механизмам выше (compound-фильтр уровня 2, relative-цвет, tagged-event на уровне 3, направленность `spawnersOf`, MechAddValue-only на уровне 5, спавнеры активаторов). На реальных данных: Бухгалтер 66→17 предметов, Военный/Казино получили всех релевантных перекрасчиков (было только по одному), Дурка 13→4 (Секретная лаборатория осталась, шум ушёл), Музей/Рок музыкант потеряли по одному тематически неродственному предмету (Рок музыкант и Дешёвый мотель соответственно), Медсестра — наоборot, получила Дешёвый мотель через цепочку спавнеров активатора (Маньяк). Регрессия зафиксирована в `relations.test.ts` (`describe("computeCascadeBuilds item selection")`) на синтетических фикстурах с реальными полями/id.
- Если на реальных данных результат снова разойдётся с ожиданиями — вероятная точка расхождения: уровню 5 всё ещё может не хватать случаев «влияния на свойства» сверх перекраски/выдачи тега/прямого MechAddValue-воздействия — пользователь сам сказал «и прочее» про исходный список, т.е. он сознательно неполный.

## Дерево связей билда (`computeBuildTree`, `buildTree.ts` + `BuildTree.tsx`)

Секция на `/builds/:id`, ниже списка предметов — визуализирует **уже существующий состав** конкретного билда (не ищет новые предметы, не дублирует автосборку), объясняя, через что каждый предмет связан с головным (`build.items[0]`).

- BFS-дерево сверху вниз: ступень 0 — головной предмет; ступень 1 — прямая связь, Card; ступень 2 — прямая связь, House/Artefact (обе ступени считаются за один и тот же первый BFS-проход, просто по типу); ступень 3+ — непрямая связь, дальше по расстоянию.
- «Прямая связь» = **сильные** сигналы `relatedItems()` (прямая ссылка по id, цепочка прокачки, produced/listened-события, и — с 2026-07-19 — то же самое тег/цвет/tagged-event сопоставление, что уже использует каскадная генерация, см. `buildCascadeStyleConnections` в `relations.ts`). Просто общий тег без механики, которая его фильтрует, связью по-прежнему не считается.
- **Пустые ступени не пропускаются** — если на ступени 1 никого нет, а на ступени 2 есть, ступень 1 всё равно рисуется, с одним пустым пунктирным слотом вместо предметов (чтобы последовательность ступеней не «скакала»).
- **Комбинации (`ReplaceItem`)** — когда `itemIdToReplace` **и** `NeededItem` оба состоят в билде (настоящая комбинация из двух ингредиентов, не просто один предмет, который во что-то превращается), они рисуются не как плоские попарные рёбра, а через синтетический узел-«⚗️»: оба ингредиента входят в него, а из него уже идёт связь к результату (`replacementItem`). Узел комбинации обнаруживается BFS-ом с той стороны, которая раньше достигнута — если результат сам является головным предметом (частый случай — комбинация как раз объясняет, откуда он взялся), связь рисуется «от корня вниз к ингредиентам», а не наоборот. Если `NeededItem` не входит в билд — это не комбинация, а обычная замена, рисуется как раньше, прямым ребром. Дедуп по набору ингредиентов+результату, а не по id правила — в реальном `ReplaceItem.csv` встречаются буквально задублированные строки для одной и той же комбинации.
  - **Цвет рёбер у комбинации** (`edgeColor` в `BuildTree.tsx`): оранжевый — ингредиент → узел комбинации, зелёный — узел комбинации → результат. Красится по реальной роли (`combo.ingredientIds`/`combo.resultId`), не по тому, что BFS записал как «родителя» — это принципиально, когда результат комбинации это сам головной предмет: он обнаруживается раньше своих ингредиентов и становится BFS-родителем узла комбинации, хотя семантически он результат (ниже по цепочке), а не источник.
- Предметы без найденной связи — отдельным блоком «Без найденной связи с головным предметом» под деревом, кликабельны, но не встроены в саму схему.
- Наведение на узел/ребро подсвечивает его и всё связанное, остальное гаснет (`neighborsOf`, построено из `parents` у всех узлов, включая узлы-комбинации).

## Граф билд↔билд (`computeBuildConnections`, `relations.ts`)

Отдельная от `relatedBuilds()` логика — специально для страницы «Граф», без «мягких» item-level сигналов:

- Ребро есть, если у двух билдов **есть общий предмет** (уровни прокачки +/++ не считаются — фильтруются через `higherTierIds`, как и везде), либо если связь **добавлена вручную**.
- Сила связи = `sharedItemCount / min(|itemsA|, |itemsB|)` — нормировка на **меньший** из двух билдов, чтобы билд из 10 предметов с 1 общим не выглядел так же сильно связан, как два билда по 2 предмета с тем же 1 общим.
- Ручные связи — поле `Build.manualLinks?: string[]`, обновляется **симметрично на обеих сторонах** через `GameStore.linkBuilds(a, b)`/`unlinkBuilds(a, b)` (не забывайте про симметрию, если трогаете эту логику руками — она осознанно продублирована на обеих записях, а не хранится как отдельный список пар). UI — секция «Связи с другими билдами» на `/builds/:id`, отдельно от вычисляемого «Возможно связано с».
- Ручные связи **не влияют** на `relatedBuilds()`/автосборку билдов — это два независимых механизма.
- Узлы графа рисуются вручную (`nodeCanvasObjectMode="replace"` в `GraphPage.tsx`): иконка/эмодзи билда в рамке (заливка `#2B2D31`, обводка `#5B8CFF`) вместо обычной точки. Обязательно нужен parные `nodePointerAreaPaint` с той же геометрией круга — иначе клики/hover перестанут работать, т.к. `react-force-graph-2d` в режиме `"replace"` использует отдельный скрытый canvas для hit-тестинга. Чекбокс «Названия» (по умолчанию включён) в левом верхнем углу переключает подписи под узлами.

- Экспорт билдов/правок обратно в Google Sheets — заявленная будущая цель пользователя, пока не реализовано. При добавлении новых моделей данных держать их сериализуемыми/generic с этим в виду.
- `CardUpgrades` не покрывает все реальные апгрейд-цепочки (пример: Черлидер/Болельщик) — это пробел в исходных данных, не в коде.

## Тестирование без реального Google Sheets

Реальные CSV пользователя лежат вне репозитория (присылались в чате). Для ручной проверки в браузере: положить CSV-файлы во временную `public/_test_data/` (не коммитить!), сформировать `File[]` через `fetch` + `DataTransfer` и продиспатчить `change`-событие на `input[type=file][accept=".csv"]` на странице «Источники» — так весь пайплайн (классификация → нормализация → GameStore) проверяется по-настоящему, без моков. Использовалось многократно в ходе разработки, детали — в истории чата/памяти агента.
